## M1
### ID
M1

### Nombre
Analizar desacople del motor de conversación respecto de WhatsApp

### Objetivo
Determinar si el diseño actual del sistema permite incorporar un nuevo canal conversacional interno (por ejemplo, una consola web o interfaz local) sin reescribir la lógica principal del flujo.

### Alcance
- relevar el estado actual del acople entre:
  - `WhatsappWebhookController`
  - `WhatsAppSender`
  - motor de conversación
  - handlers
  - `ConversationFlowResolver`
  - `StepResult`
  - `MessageResolver`
- identificar qué partes dependen todavía del formato o comportamiento de WhatsApp
- analizar si hace falta:
  - DTO interno de mensaje
  - adapter de canal
  - abstracción de canal de salida
- producir un diagnóstico técnico y una recomendación concreta

### No incluye
- implementación de consola web
- implementación de Telegram
- refactor masivo del motor
- frontend
- API nueva funcionando

### Dependencias
- ninguna

### Validación automática obligatoria
- no aplica test obligatorio si el resultado es solo análisis/documentación
- si se modifican archivos de código, correr:
  - `make test`

### Validación manual sugerida
- revisar que el análisis deje claro:
  - fortalezas del diseño actual
  - acoples al canal
  - cambios mínimos necesarios
  - recomendación de esfuerzo aproximado

### Stop conditions específicas
- si el análisis requiere confirmar decisiones de negocio no documentadas, marcar `needs_review`
- si para responder hace falta una refactorización de código no prevista, frenar y dejarla propuesta, no implementarla

### Entregable esperado
- diagnóstico técnico claro
- propuesta de arquitectura mínima para soportar otro canal
- actualización documental mínima si realmente aporta
- backlog de hallazgos fuera de alcance si aparecen

### Estado
`pending`

## M2
### ID
M2

### Nombre
Definir plan técnico corto para una consola/interfaz local de chat

### Objetivo
A partir del análisis del M1, dejar una propuesta concreta y acotada para implementar una primera consola/interfaz local de chat contra el motor de conversación.

### Alcance
- definir milestones chicos para una futura implementación
- decidir si conviene empezar por:
  - adapter/DTO interno
  - API/controlador interno
  - canal local de prueba
  - UI mínima
- dejar la secuencia sugerida de implementación
- documentar qué entra en una primera versión y qué no

### No incluye
- implementación completa de la consola
- frontend final
- tiempo real / websockets
- autenticación
- multicanal real funcionando

### Dependencias
- M1 debe estar `done` o `needs_review` con definición suficientemente clara

### Validación automática obligatoria
- no aplica test obligatorio si el resultado es solo documentación/plan
- si se toca código:
  - `make test`

### Validación manual sugerida
- revisar que el plan quede dividido en milestones chicos, revisables y con commits acotados

### Stop conditions específicas
- si M1 no deja una arquitectura mínima clara, no avanzar
- si aparecen dos enfoques válidos y no hay suficiente definición, frenar y dejar opciones explícitas

### Entregable esperado
- plan técnico corto para la futura consola
- milestones sugeridos
- recomendación de enfoque inicial
- actualización de `plan_dev/BACKLOG.md` si surgen tareas futuras

### Estado
`pending`

## M3
### ID
M3

### Nombre
Mejorar trazabilidad y logs de conversación

### Objetivo
Identificar y aplicar mejoras chicas y útiles en logs y trazabilidad para facilitar debug, soporte y futura consola interna.

### Alcance
- revisar logs actuales del flujo conversacional
- detectar huecos de trazabilidad
- mejorar naming o contenido de logs si hace falta
- asegurar que se pueda seguir mejor una conversación de punta a punta
- si es razonable, dejar una convención más clara para logs conversacionales

### No incluye
- rediseño total de logging
- observabilidad enterprise
- integración con plataformas externas de monitoreo
- anonimización completa de PII en este milestone si eso requiere análisis mayor

### Dependencias
- idealmente después de M1, porque el análisis de canal puede revelar puntos de trazabilidad importantes

### Validación automática obligatoria
- `make test` si se modifica lógica relevante o tests relacionados

### Validación manual sugerida
- revisar que desde logs/eventos se pueda entender:
  - qué conversación se procesó
  - qué paso transitó
  - qué error ocurrió
  - qué salida se envió

### Stop conditions específicas
- si la mejora de logs deriva en una revisión grande de protección de datos, frenar y mandar ese tema a backlog
- si el cambio deja de ser pequeño y acotado, cortarlo y dejar propuesta

### Entregable esperado
- mejoras chicas de trazabilidad/logs
- documentación mínima si conviene
- backlog de temas mayores detectados

### Estado
`pending`
