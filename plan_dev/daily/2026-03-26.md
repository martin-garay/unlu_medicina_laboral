## M1
### ID
M1

### Nombre
Analizar desacople del motor de conversación respecto de WhatsApp

### Objetivo
Determinar si el diseño actual del sistema permite incorporar un nuevo canal conversacional interno (por ejemplo, una consola web o interfaz local) sin reescribir la lógica principal del flujo.

### Alcance
- relevar el estado actual del acople entre:
  - `WhatsappWebhookController`
  - `WhatsAppSender`
  - motor de conversación
  - handlers
  - `ConversationFlowResolver`
  - `StepResult`
  - `MessageResolver`
- identificar qué partes dependen todavía del formato o comportamiento de WhatsApp
- analizar si hace falta:
  - DTO interno de mensaje
  - adapter de canal
  - abstracción de canal de salida
- producir un diagnóstico técnico y una recomendación concreta

### No incluye
- implementación de consola web
- implementación de Telegram
- refactor masivo del motor
- frontend
- API nueva funcionando

### Dependencias
- ninguna

### Validación automática obligatoria
- no aplica test obligatorio si el resultado es solo análisis/documentación
- si se modifican archivos de código, correr:
  - `make test`

### Validación manual sugerida
- revisar que el análisis deje claro:
  - fortalezas del diseño actual
  - acoples al canal
  - cambios mínimos necesarios
  - recomendación de esfuerzo aproximado

### Stop conditions específicas
- si el análisis requiere confirmar decisiones de negocio no documentadas, marcar `needs_review`
- si para responder hace falta una refactorización de código no prevista, frenar y dejarla propuesta, no implementarla

### Entregable esperado
- diagnóstico técnico claro
- propuesta de arquitectura mínima para soportar otro canal
- actualización documental mínima si realmente aporta
- backlog de hallazgos fuera de alcance si aparecen

### Estado
`done`

### Resultado de ejecución
- el núcleo conversacional actual ya es reutilizable en buena parte: `ConversationFlowResolver`, handlers, validadores, `StepResult`, `MessageResolver`, servicios de conversación y servicios de negocio no dependen directamente de `Request` ni del payload crudo de Meta
- el acople principal sigue en los bordes: `WhatsappWebhookController`, `WhatsAppSender` y `ConversationTimeoutService`
- hoy es viable sumar un canal interno sin reescribir handlers, pero no conviene hacerlo sin extraer antes una capa mínima de orquestación y una abstracción de salida por canal
- se actualizó `docs/05-motor-de-conversacion.md` con el diagnóstico y la recomendación técnica mínima

## M3
### ID
M3

### Nombre
Extraer DTO interno y servicio de interacción conversacional

### Objetivo
Implementar la capa mínima común que permita desacoplar la interacción conversacional del canal WhatsApp sin reescribir handlers ni servicios de negocio.

### Alcance
- definir DTO o estructura interna de mensaje entrante
- definir DTO o estructura interna de respuesta saliente
- extraer un servicio tipo `ConversationInteractionService`
- mover a esa capa la orquestación central hoy repartida entre webhook, aplicación de `StepResult` y registro básico de trazabilidad
- mantener comportamiento funcional equivalente para el canal WhatsApp existente

### No incluye
- controller interno nuevo
- UI de consola
- rediseño total de persistencia
- reescritura de handlers

### Dependencias
- M2 `done`

### Validación automática obligatoria
- `make test`
- si existe un set de tests más acotado para conversación, puede correrse adicionalmente

### Validación manual sugerida
- revisar que la nueva capa pueda recibir una entrada normalizada y devolver una salida estructurada sin depender de `Request`

### Stop conditions específicas
- si la extracción obliga a reescribir masivamente handlers o validadores, frenar y acotar
- si aparecen dos diseños razonables para el DTO y no hay criterio suficiente, documentar opciones y dejar `needs_review`

### Entregable esperado
- servicio común de interacción conversacional
- DTOs o estructuras internas mínimas
- base lista para adaptar WhatsApp y un canal interno

### Estado
`done`

### Resultado de ejecución
- se agregaron `ConversationInboundMessage`, `ConversationOutboundMessage` y `ConversationInteractionResult` como DTOs/estructuras mínimas de entrada y salida
- se creó `ConversationInteractionService` para concentrar la orquestación central del flujo conversacional
- el servicio nuevo ya resuelve conversaciones por `channel + participant_id`, evitando mezclar el canal interno con WhatsApp
- se ajustó `ConversationManager` para soportar lookup y alta por canal sin romper el contrato existente de WhatsApp
- la base quedó lista para que WhatsApp y un canal interno reutilicen el mismo núcleo

## M4
### ID
M4

### Nombre
Adaptar WhatsApp y timeout service a la nueva capa común

### Objetivo
Hacer que el canal WhatsApp existente utilice la nueva capa de interacción sin cambiar el comportamiento del flujo.

### Alcance
- reducir `WhatsappWebhookController` a parseo del request y delegación
- adaptar el envío saliente detrás de una abstracción por canal o equivalente
- desacoplar `ConversationTimeoutService` de `WhatsAppSender`
- preservar trazabilidad y comportamiento actual

### No incluye
- nuevo canal interno todavía operativo
- rediseño de menús o UX del bot
- cambios funcionales en aviso o anticipo

### Dependencias
- M3 `done`

### Validación automática obligatoria
- `make test`

### Validación manual sugerida
- revisar que el flujo actual de WhatsApp siga respondiendo igual en texto, menú y timeouts

### Stop conditions específicas
- si el cambio rompe contratos actuales de handlers o servicios de negocio, frenar
- si el desacople del timeout service exige una refactorización mayor no prevista, cortar y documentar

### Entregable esperado
- webhook reducido a adapter
- timeout service desacoplado del sender concreto
- canal principal todavía funcional sobre la nueva base

### Estado
`done`

### Resultado de ejecución
- `WhatsappWebhookController` quedó reducido a parseo, normalización y delegación sobre `ConversationInteractionService`
- se incorporó la interfaz `ConversationChannelSender` y el router `ConversationChannelRouter` para encapsular el envío saliente por canal
- `ConversationTimeoutService` dejó de depender directamente de `WhatsAppSender` y pasó a usar la abstracción común de envío
- el comportamiento del canal WhatsApp se preservó y quedó cubierto por la suite
