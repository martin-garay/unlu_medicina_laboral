# Plan de desarrollo - Fase 2
## Fecha
2026-03-15

## Objetivo

Este documento organiza los próximos pasos de implementación sobre la base ya construida del motor de conversación.

Se asume que la etapa previa ya dejó implementado:

- tablas y modelos de conversaciones, mensajes y eventos
- servicios base del motor de conversación
- integración mínima del webhook con trazabilidad
- base suficiente para comenzar a desacoplar el flujo actual

La intención de este plan es que sirva como guía para los próximos prompts de desarrollo, permitiendo avanzar en cambios pequeños, revisables y orientados a commits claros.

---

## Estado actual asumido

A esta altura del proyecto ya debería existir una base que permita:

- crear o recuperar una conversación activa
- registrar mensajes entrantes y salientes
- registrar eventos de conversación
- asociar el tráfico del webhook a una conversación
- mantener trazabilidad básica sin depender todavía de los flujos finales

Todavía no se considera resuelto en esta etapa:

- estructura formal por pasos
- mensajes centralizados
- parámetros de negocio centralizados
- menú principal real
- identificación común reutilizable
- flujo completo de aviso o certificado
- scheduler de inactividad
- validaciones avanzadas por paso

---

## Objetivo de esta fase

Comenzar a transformar el flujo actual en una arquitectura extensible, desacoplada y mantenible, avanzando desde:

- textos y parámetros centralizados
- estructura base de flujos por pasos
- refactor gradual del controller actual
- implementación del primer flujo real
- identificación común reutilizable

---

# Paso 1
## Centralizar textos y parámetros

### Objetivo
Sacar del código los textos y configuraciones que hoy estén hardcodeados, para dejar preparada una base más mantenible y lista para crecer.

### Alcance
Crear o completar:

- `lang/es/whatsapp.php`
- `config/medicina_laboral.php`
- primeros templates Blade para mensajes largos

### Qué debe incluir `lang/es/whatsapp.php`
Una estructura clara y mantenible para textos del chatbot, por ejemplo:

- `general`
- `menu`
- `identificacion`
- `aviso`
- `certificado`
- `errores`
- `interacciones_invalidas`
- `timeouts`
- `cancelacion`
- `confirmaciones`
- `derivacion`

### Qué debe incluir `config/medicina_laboral.php`
Una base de parámetros y catálogos configurables, por ejemplo:

#### conversation
- cantidad máxima de intentos inválidos
- palabras clave de cancelación
- palabras clave de reinicio
- umbrales de inactividad

#### certificados
- cantidad máxima de archivos
- extensiones permitidas
- mime types permitidos
- tamaño máximo
- plazo de carga del anticipo

#### catalogos
- sedes
- tipos de ausentismo
- tipos de certificado
- parentescos si aplica

### Qué templates Blade deben crearse
Primeros templates de mensajes largos o institucionales, por ejemplo:

- confirmación final de aviso
- aviso registrado
- confirmación final de anticipo
- anticipo registrado
- cancelación de aviso
- cancelación de certificado
- mensajes de inactividad si resulta útil

### Consideraciones para el agente
- no reescribir todavía toda la lógica del flujo
- no meter lógica de negocio en Blade
- no dejar strings largos institucionales hardcodeados
- puede hacer refactor mínimo seguro si ayuda a empezar a usar `__()` y `config()`
- cuando existan claves temporales para sostener el flujo actual, preferir nomenclatura `transicional` y evitar `legacy`

### Entregable esperado
Una base clara de mensajes y configuración, lista para ser usada por los pasos siguientes.

# Paso 2
## Definir la estructura extensible de flujo

### Objetivo
Diseñar la base técnica para manejar el flujo conversacional paso a paso, evitando seguir creciendo con lógica distribuida en el controller.

### Alcance
Crear la estructura mínima para una arquitectura de flujo extensible.

### Componentes esperados

#### `ConversationFlowResolver`
Responsable de decidir qué flujo o handler corresponde según el estado actual de la conversación.

#### `StepResult`
Objeto de resultado del procesamiento de un paso.

Debe poder representar al menos:
- si el input fue válido o inválido
- siguiente paso
- mensaje a enviar
- código de error si corresponde
- si incrementa intentos
- si hay cambio de estado
- si corresponde cancelar o finalizar

#### `StepHandler`
Interfaz o clase base para procesar un paso del flujo.

Cada handler debería poder:
- saber qué paso representa
- procesar input
- devolver un `StepResult`

#### `Validator`
Interfaz o clase base para validar un dato de entrada.

La validación no debería:
- enviar mensajes
- persistir datos directamente
- mezclar texto final con lógica

Debería devolver resultados estructurados o códigos de error.

#### `MessageResolver`
Responsable de traducir resultados de validación o de flujo a mensajes concretos, apoyándose en:

- `lang/es/whatsapp.php`
- templates Blade cuando corresponda

### Consideraciones para el agente
- mantener la solución simple y no sobrediseñada
- no construir todavía toda la máquina de estados completa
- no intentar resolver todos los pasos reales de una sola vez
- priorizar contratos e implementación base usable

### Entregable esperado
Una base estructural clara que permita mover el flujo actual hacia pasos desacoplados.

---

# Paso 3
## Refactor mínimo del flujo actual

### Objetivo
Reducir la dependencia del `switch` actual dentro del controller y empezar a mover el procesamiento hacia la nueva estructura extensible.

### Alcance
Hacer un refactor mínimo y seguro del flujo actual para que ya no dependa tanto del controller.

### Qué se espera
- mantener el comportamiento actual del MVP en lo posible
- mover responsabilidades fuera del controller
- comenzar a usar:
  - `ConversationFlowResolver`
  - `StepHandler`
  - `StepResult`
  - `MessageResolver`

### Qué no se espera todavía
- implementar todos los pasos reales del negocio
- resolver todas las validaciones finales
- rehacer completamente el bot desde cero
- introducir scheduler en este punto

### Consideraciones para el agente
- el controller debe quedar más liviano
- el refactor debe ser gradual
- si conviene mantener partes del comportamiento actual temporalmente, hacerlo y explicarlo
- no romper el flujo mínimo existente

### Entregable esperado
El webhook empieza a delegar en una estructura más limpia y la lógica deja de estar tan concentrada en un `switch` grande.

# Paso 4
## Implementar el primer subflujo real

### Objetivo
Construir el primer flujo conversacional real sobre la nueva base estructural.

### Alcance
Implementar:

- menú principal
- selección entre aviso de ausencia y anticipo de certificado
- opción cancelar y volver al menú

### Qué debe contemplar el menú principal
Opciones visibles o equivalentes para:

1. Consultas
2. Aviso de ausencia
3. Anticipo de certificado médico

### Prioridad funcional
Aunque el documento contempla “Consultas”, en esta fase el foco sigue estando en:

- aviso de ausencia
- anticipo de certificado médico

La opción de consultas puede quedar:
- visible pero no implementada
- reservada para más adelante
- respondida con un mensaje de “no disponible aún”, si resulta razonable

### Qué debe contemplar la cancelación
La opción “Cancelar y volver al menú” debe:

- cerrar lógicamente el flujo en curso si corresponde
- no borrar la conversación previa
- permitir continuar a partir de una nueva conversación o reinicio controlado, según el diseño adoptado
- registrar trazabilidad suficiente

### Consideraciones para el agente
- apoyarse en la estructura creada en el paso 2
- no intentar implementar todavía todo el flujo de aviso ni de certificado
- concentrarse en navegación y selección de flujo

### Entregable esperado
El sistema ya tiene un punto de entrada conversacional real y ordenado, basado en la nueva estructura.

---

# Paso 5
## Implementar la identificación común

### Objetivo
Construir el bloque reutilizable de identificación del trabajador para ambos flujos principales.

### Datos a solicitar
- nombre
- legajo
- sede
- jornada laboral

### Alcance
Implementar estos pasos como parte común y reutilizable del flujo.

### Persistencia transitoria
La información ingresada en esta etapa debe persistirse de forma transitoria en la conversación.

### Opciones de persistencia transitoria
Dependiendo de la base actual, puede resolverse inicialmente mediante:

- `metadata` de la conversación
- estructura transitoria equivalente ya existente

No es obligatorio crear todavía una tabla específica para respuestas por paso, salvo que resulte realmente necesario.

### Consideración funcional importante
La identificación real del trabajador todavía puede quedar:

- mockeada
- desacoplada detrás de una interfaz o servicio
- validada solo de forma básica en esta etapa

### Qué debería contemplar este bloque
- solicitud ordenada de cada dato
- validaciones mínimas razonables
- persistencia de respuestas parciales
- posibilidad de reutilización en aviso y certificado

### Consideraciones para el agente
- no mezclar todavía identificación común con validaciones externas complejas
- no acoplar esta etapa a SIU-Mapuche ni a otro sistema real
- dejar la base preparada para que luego pueda reemplazarse por un servicio real

### Entregable esperado
Un subflujo común de identificación que pueda usarse luego como entrada compartida para aviso de ausencia y anticipo de certificado.

## Orden recomendado de ejecución

Se recomienda avanzar exactamente en este orden:

1. centralización de textos y parámetros
2. estructura extensible de flujo
3. refactor mínimo del flujo actual
4. menú principal real
5. identificación común

Este orden permite evitar que:
- la estructura de pasos nazca con textos hardcodeados
- el controller siga creciendo desordenadamente
- el flujo real se implemente antes de tener una base reutilizable

---

## Qué no abordar todavía en esta fase

Todavía no corresponde avanzar con:

- scheduler e inactividad automática
- flujo completo de aviso de ausencia
- flujo completo de anticipo de certificado
- adjuntos finales completos
- emails
- integración real con sistemas externos
- adapter formal del payload externo
- validaciones avanzadas de negocio

Eso debería venir después de completar esta fase.
